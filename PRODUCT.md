# Lite Annotate Product Context

## Register

product

## Product Purpose

Lite Annotate turns in-app customer bug reports into durable engineering memory, diagnosis, verification, and scoped fix PRs.

## Primary Users

- Developer founders who need fast, trustworthy bug triage without manually reconstructing every report.
- Engineers reviewing captured reports, memory context, diagnosis evidence, verification output, and PR readiness.

## Secondary Users

- Product users reporting a bug from inside a customer app through the embedded widget.

## Brand Tone

Sharp, technical, evidence-led, calm. The product should feel like an engineering command center, not a marketing page.

## Strategic Principles

- Evidence before action: every diagnosis and PR must show browser, memory, code, and verification receipts.
- Trust over spectacle: the UI should make risky actions clear and gated.
- Memory is the differentiator: show how native GBrain changes the agent from cold start to context-aware.
- Fast demo comprehension: a hackathon judge should understand the capture to memory to fix loop within one screen.

## Anti-References

- Generic SaaS landing pages with oversized hero metrics.
- AI dashboards filled with decorative gradients and repeated cards.
- Dense raw JSON dumps as the primary way to understand state.
- Unlabeled destructive or externally visible actions.

## Success Criteria

- Demo page makes the user report flow obvious.
- Dashboard makes report status scannable.
- Report detail explains memory impact and analysis state before raw payloads.
- Dry run analysis is visually safer than PR-opening analysis.
- Raw JSON remains available for engineers but no longer dominates the first viewport.
